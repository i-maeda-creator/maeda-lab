HTTP ブラウザとサーバーが交わす、注文票の形式です。

HTTPはWebの会話です。どのURLへ、どのメソッドで、どんなヘッダと本文を送り、どんなステータスで返ってきたかを読みます。

カフェの注文画面で「注文する」を押すと、ブラウザはサーバーへHTTPリクエストを送ります。サーバーは「成功しました」「ログインしてください」「権限がありません」「内部で失敗しました」といった結果をHTTPレスポンスで返します。HTTPは画面の裏にある、かなりきっちりした伝票のやり取りです。

HTTPリクエストの中身

HTTPでは、メソッド、パス、ヘッダ、本文を見ます。GETは取得、POSTは作成や実行、PUT/PATCHは更新、DELETEは削除に使われます。ヘッダにはCookie、Authorization、Content-Type、Acceptなどが入り、ログイン状態や本文形式を伝えます。本文にはJSONやフォームデータが入ります。

ステータスコードを読む

分類意味カフェの例
2xx成功注文登録、メニュー取得に成功
3xxリダイレクトログイン画面へ戻される
400リクエスト不正JSON形式が壊れている
401 / 403認証/認可の問題未ログイン、権限不足、WAF遮断
415本文形式が不適切Content-Typeが違う
500 / 502 / 504サーバー側または中継の失敗アプリ例外、LBからバックエンドタイムアウト

実務でよく見る切り分け

「ブラウザでは動くがAPIクライアントでは失敗する」なら、Content-Type、Accept、Cookie、Authorization、CSRFトークンを比較します。「GETは成功するがPOSTだけ失敗する」なら、本文サイズ、WAF、LB制限、アプリ処理時間を見ます。HTTPはL7ですが、背後にはTLS、LB、FW、アプリ、DBがいるため、ステータスコードを手がかりに層を絞ります。

Quick Quiz 1

注文一覧を取得する操作に最も近いHTTPメソッドはどれでしょうか。

取得はGETです。作成はPOST、更新はPUT/PATCH、削除はDELETEが代表です。

Applied Quiz 2

ログイン後の管理画面だけ403になります。まず見るべきものはどれでしょうか。

403は権限やアクセス制御の問題です。ログイン済みか、許可されているか、WAFで止められていないかを見ます。

Network Specialist Written

HTTP 午後記述

問題の背景

カフェ注文APIで、ブラウザ操作は成功しますが、外部連携用のAPIクライアントから送ると 415 Unsupported Media Type になります。別件で、GET /menu は200ですが、POST /orders は本文サイズが大きいと504になります。LBのヘルスチェックはGET /healthで常に200、WAFログにはPOST /ordersのJSON本文に対する検知が一部あります。アプリログでは、成功時は注文IDが発行されますが、失敗時はアプリまで到達していないリクエストもあります。

短答記述

ブラウザでは成功し、APIクライアントでは415になる場合、HTTPの観点で比較すべき要素を四つ挙げてください。

事例ベースの午後問題

GETは成功するがPOSTだけ失敗する障害について、HTTPメソッド、Content-Type、本文サイズ、認証ヘッダ、WAF、LBヘルスチェック、アプリ到達性の観点で切り分け手順を説明してください。ヘルスチェック200をもって注文APIが正常とは判断できない理由も述べてください。

回答はこちらをクリック

模範解答・採点観点

  • メソッド、Content-Type、Accept、Authorization、Cookie、CSRFトークン、本文形式を比較対象に挙げる。
  • 415は本文の種類やContent-Type不一致で起きるため、application/jsonか、フォームか、multipartかを確認する。
  • POSTだけ504なら、本文サイズ、WAF検知、LBのリクエスト/アイドルタイムアウト、バックエンド処理時間を見る。
  • GET /healthは短く単純な通信であり、POST /ordersの本文、認証、WAF、DB更新を代表しないと説明する。
  • LBログ、WAFログ、アプリログを相関IDや時刻で突き合わせ、どこまで到達しているかを確認する。

解説

HTTPの午後問題では、ステータスコードを暗記ではなく証拠として読みます。415なら、サーバーがリクエスト本文の形式を受け入れられないという意味です。ブラウザとAPIクライアントで成功/失敗が分かれるなら、送っているヘッダや本文が違う可能性が高くなります。

一方、POSTだけ504になる場合は、アプリのバグとは限りません。WAFで本文が止まる、LBがバックエンド応答を待ち切れない、FWやTCPのタイムアウトに当たる、アプリまで届く前に落ちる、など複数の層が絡みます。ヘルスチェックが200でも、それは「GET /healthが返る」ことを示すだけです。実際の注文と同じメソッド、同じヘッダ、同じ本文サイズで再現することが重要です。
