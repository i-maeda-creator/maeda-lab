01-6 / Network

Cafe Network Flow カフェで注文画面が開くまでを、最初から最後まで追います。

L1からL7までを別々に学んだあと、最後に一つの通信としてつなげます。 カフェのレジ端末がクラウドの注文管理画面を開くまでを、現場の切り分け目線で見ていきます。

朝のカフェです。スタッフがレジ端末を起動し、注文管理画面を開きます。 画面にはログインフォームが出るだけですが、その裏ではケーブル、Wi-Fi、スイッチ、ルーター、DNS、TLS、HTTP、サーバーが順番に働いています。

ネットワークの面白いところは、どれか一つが欠けると「ただ開かない」に見えることです。 だからこそ、層ごとに何が起きているかを分けて見ます。

全体像を一枚で見ましょう

1. まずL1です。信号が届かなければ何も始まりません

レジ端末がWi-Fiでつながるなら電波、APがPoEで動くならLANケーブルと給電、店舗から外へ出るならONUや光回線が関わります。 電源、ケーブル、MDF、ONU、APのランプ。地味ですが、最初にここを見ないと遠回りになります。

2. 次にL2です。同じ店内で誰に渡すかを決めます

APからスイッチへ入り、レジ端末の通信は業務用VLANへ流れます。 客用Wi-Fiとレジ端末が同じネットワークにいると危険なので、VLANで分けます。 ここでポート設定やタグVLANを間違えると、電波は強いのに業務システムだけ見えない、という状態になります。

3. L3では、店の外へ出る道を選びます

レジ端末は宛先が店内ではないと判断し、デフォルトゲートウェイへ送ります。 ルーターはNATを行い、必要ならVPNや閉域網、インターネット回線へ流します。 ここではIPアドレス、サブネット、ルート、NAT、FWの責任分界点を見ます。

4. L4では、クラウド側のどの窓口へ行くかを決めます

注文管理画面なら多くの場合TCP 443番です。 FWで443番が許可され、LBのリスナーが受け、正常なWebサーバーへ振り分けます。 pingが通っても443番が閉じていれば、ブラウザでは「開かない」に見えます。

5. L5-7では、人が見る画面の話になります

DNSでドメイン名をIPに変換し、TLSで証明書を確認し、HTTPでリクエストを送ります。 ログインではCookie、セッション、認証基盤、API Gateway、DB接続が関係します。 ここまで来ると、ネットワーク担当、アプリ担当、認証基盤担当、クラウド担当が一緒に調べることも多いです。

切り分けのコツは、質問を層ごとに変えることです

見え方まず考える層確認すること
APのランプが消えているL1電源、PoE、ケーブル、AP故障
客用Wi-Fiは見えるがレジだけ社内へ行けないL2VLAN、SSID、スイッチポート
店内プリンタは見えるがクラウドへ行けないL3ゲートウェイ、NAT、上流回線、ルート
pingは通るがHTTPSだけ失敗L4TCP 443、FW、LB、待受ポート
ログイン後だけ403L7認可、Cookie、WAF、セッション

ネットワークスペシャリスト級まとめテストです

ここからは一気通貫の切り分け問題です。 正解以外もそれっぽくしています。ログ、到達範囲、層の違いを見て判断してください。

このページの落ち

通信は、一本の線ではなく、たくさんの担当者がリレーする駅伝のようなものです。 どこかで止まったら、怒るより先に、どの層で止まったかを見ます。 それができると、「ネットが悪いです」が、かなり具体的な調査に変わります。

Network Specialist Written

総合午後問題

短答記述

注文アプリだけ失敗します。Wi-Fi、DNS、HTTPS、API、DBのどこから切り分けるか順序を答えてください。

事例ベースの午後問題

来客Wi-Fiではトップは表示されますが決済だけ失敗します。従業員端末では成功します。AP、VLAN、FW/NAT、DNS、CDN、WAF、LB、アプリ、決済SaaSを含め、L1からL7で原因候補を説明してください。

模範解答・採点観点

  • 来客VLANのFW/NAT、DNS、WAF、決済SaaS到達、Cookie/認証、TLSを挙げる。
  • L1/L2/L3/L4/L7の順に観測点を置き、成功/失敗経路を比較する。
  • トップ表示成功から静的配信は生き、決済APIや外部連携が怪しいと推論する。
